Machines can read the code now.
The code still does not say why.

Every modernisation pitch is sold on reading old code. That capability is now a commodity: the best tool and a general-purpose model are one point apart on the industry benchmark. The thing that actually breaks the project was never in the code to begin with, and no extraction tool can recover it.

Ruling causes in and out

Three findings, in the order a diagnosis has to take them.

000210

Machines can read the code now

Extraction is a commodity

On LegacyCodeBench, Legacy Insights scores 94, Claude Sonnet 93, Opus 91, AWS Transform 90. The specialist tool leads a general-purpose model by a single point, and in the January 2026 release of the same benchmark Claude Sonnet beat it on the hardest tier, 92 to 90. The benchmark was built by Kalmantic with Hexaview, whose own product tops it. Extraction is not where the differentiated value sits.

000220

But the code does not say why

The reasoning was never written down

A tool can tell you that a routine applies a 4.5 percent adjustment on a particular branch. It cannot tell you that the figure came from a 1998 regulator settlement, that it was meant to expire, or which of three business owners agreed to it. That was decided in a meeting, and the meeting was not the artefact. The people who remember it are the ones now retiring.

000230

And the usual fix rebuilds the same problem

Java is the standard destination

COBOL to Java is the industry default. In Veracode’s Spring 2026 update, Java scored 29 percent on secure code generation, the worst of the four languages tested, because models trained heavily on legacy Java reproduce the insecure idioms that corpus contains. A migration that converts an unverified specification into a language the model writes worst is not modernisation, it is a re-run.

000300

The chain

Six steps. Only the first is a cause. The other five are consequences that get reported as the cause.

  1. Cause

    The reasoning behind a decision was never recorded. Only its implementation was.

  2. Consequence 01

    So every change re-derives that behaviour by hand, from the code, one engineer at a time.

  3. Consequence 02

    So the re-derivation is unverifiable. Nobody can confirm the reading is right, because there is nothing to check it against.

  4. Consequence 03

    So teams avoid touching the parts they cannot verify, and route around them instead.

  5. Consequence 04

    So untouched code accumulates, and gets reported upward as technical debt.

  6. Consequence 05

    So the migration converts a specification nobody verified, and fails for reasons the post-mortem records as complexity.

Read the chain backwards and the standard remedies all attach to consequences. Buying a better extraction tool attacks step one of the wrong sequence. Rewriting in a modern language attacks step five. Neither touches the cause, which is why the failure rate has not moved despite the tooling getting dramatically better.

000400

What the firm should sell

Buy the reading. Sell the record. Put a name on the sign-off.

Extraction is bought, not built, because it is a commodity input and the market is already competing it to zero. What the client keeps afterwards is the asset: a verified, attributed record of why each rule exists, who confirmed it, and on what date. That record is what makes the next change cheap, and it is the only artefact that survives the retirement of the people who remember.

Governance is the licence to operate. Someone has to sign that a reading of a rule is correct, and be accountable for it, or the record is just another undocumented assumption with better formatting.

The arithmetic that makes this a business-model question, not a delivery question: the published effort reduction on this work is around 70 percent. Billing that scope by time and materials means deleting roughly 0.70 of your own revenue on it. A retainer against the maintained record only works where the retainer is smaller than the volume of change requests multiplied by what re-deriving the reasoning costs each time. That is a condition to be tested per client, not a number to be asserted.

Claims withdrawn during the audit
